# Memory System

Squad0's memory system is designed around cognitive psychology principles. It doesn't just store and retrieve — it forgets, strengthens, and consolidates like an actual mind.

## Architecture

Each agent has a personal SQLite database (`data/agents/{name}.db`) containing:

- **Entities**: named things the agent knows about (modules, files, patterns, tools, concepts)
- **Relationships**: directed edges between entities (depends_on, breaks_when, pairs_with, replaces)
- **Facts**: specific learnings tied to entities, with confidence scores
- **Beliefs**: causal beliefs that evolve with evidence
- **Episodes**: session logs with embeddings for semantic search

A shared project database (`data/project.db`) holds cross-cutting knowledge accessible to all agents.

## MCP Memory Tools

Agents interact with their memory during sessions via five MCP tools:

### `recall`
Search memory for knowledge relevant to a topic. **Mandatory at the start of every session.**

```
recall(query: "payments module error handling")
→ Facts: [warning, 0.9] Stripe webhooks need idempotency keys
→ Beliefs: [0.7] always add timeout handling to external API calls
→ Past sessions: [success] SQ-42 — fixed payment retry logic
```

### `remember_fact`
Store a specific fact about a module, file, or concept. **Use immediately when you discover something important — don't wait.**

```
remember_fact(entity: "payments", content: "Stripe API has a 100 req/s rate limit", type: "warning")
```

### `store_belief`
Store a principle developed from experience. Beliefs evolve over time as they're confirmed or contradicted.

```
store_belief(content: "integration tests catch more real bugs than mocks in this codebase")
```

### `note_entity`
Record that you've encountered a module, file, or concept. Creates if new, updates if it exists.

```
note_entity(name: "auth", type: "module", summary: "JWT-based authentication with refresh tokens")
```

### `recall_entity`
Get everything you know about a specific entity — its facts, connections, and related entities.

```
recall_entity(name: "auth")
→ auth (module) — JWT-based authentication
→ Known facts: [observation, 0.8] session tokens expire after 1 hour
→ Connected to: database (module), middleware (pattern)
```

## Hybrid Search

Retrieval combines two strategies:

1. **Vector similarity** via brute-force cosine similarity over serialised float32 embeddings (generated by Ollama's `nomic-embed-text` model)
2. **BM25 keyword search** via SQLite FTS5 indexes

Scores are fused: `finalScore = vectorWeight × vectorScore + keywordWeight × keywordScore`

## Cognitive Psychology Principles

### 1. Forgetting Is a Feature

Beliefs decay over time using exponential decay:

```
retrieval_score = confidence × exp(-decay_rate × days_since_confirmed)
```

Old unconfirmed beliefs naturally drop from top-K retrieval without being deleted. This prevents cognitive paralysis from thousands of irrelevant memories competing for attention.

### 2. Retrieval Strengthens Memory

Every time an agent recalls and uses a memory, that memory gets a boost. The act of retrieving strengthens the memory (the testing effect). Memories that are frequently useful become more prominent. Memories that are stored but never retrieved naturally fade.

### 3. Emotional Salience — Pain Teaches More

Beliefs formed from failed sessions get higher initial confidence (1.4x) than beliefs from successful sessions. A rejected PR, a failed deployment, a bug that slipped through — these create stronger memories because the cost of repeating them is higher.

### 4. Consolidation During Personality Regeneration

The periodic personality regeneration (every N sessions) is the "sleep" phase. During consolidation:
- Related beliefs are clustered into higher-level understanding
- Contradictions between beliefs are resolved or flagged
- Repeated patterns across tickets are extracted into general principles

### 5. Context-Scoped Retrieval

Retrieval is scoped by the current ticket's context — modules, files, and concepts mentioned in the ticket description weight memory results. Working on a payments ticket surfaces payment-related memories, not auth memories.

## Fallback Extraction

If an agent runs out of context or crashes, the orchestrator parses the session transcript and extracts learnings externally. This is a safety net, not the primary path — agents should actively manage their own memory using the MCP tools.
