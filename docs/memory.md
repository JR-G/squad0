# Memory System

Squad0's memory system is designed around cognitive psychology principles. It does not just store and retrieve — it forgets, strengthens, and consolidates.

## Architecture

Each agent has a personal SQLite database (`data/agents/{role}.db`) containing:

- **Entities**: named things the agent knows about (modules, files, patterns, tools, concepts)
- **Relationships**: directed edges between entities (`depends_on`, `breaks_when`, `pairs_with`, `replaces`) with confidence scores and temporal validity
- **Facts**: specific learnings tied to entities, with confidence, confirmation count, and access tracking
- **Beliefs**: causal beliefs that evolve with evidence — confirmations increase confidence, contradictions decrease it
- **Episodes**: session logs with embeddings for semantic search

A shared project database (`data/project.db`) holds cross-cutting knowledge accessible to all agents.

### Schema Migrations

The database uses a versioned migration system. Migration 1 creates the initial schema (entities, relationships, facts, episodes, beliefs, FTS5 indexes). Migration 2 adds cognitive memory fields (`last_accessed_at`, `access_count`, `source_outcome`).

## MCP Memory Tools

Agents interact with their memory during sessions via five MCP tools exposed by the `squad0-memory-mcp` binary:

| Tool | Purpose |
|------|---------|
| `recall` | Search memory for knowledge relevant to a query. **Mandatory at session start.** |
| `remember_fact` | Store a fact about an entity (observation, preference, warning, technique) |
| `store_belief` | Store a belief or principle developed from experience |
| `note_entity` | Record an entity (module, file, pattern, tool, concept) — creates or updates |
| `recall_entity` | Get everything known about a specific entity (facts, connections, related entities) |

The MCP server runs as a child process per session, configured via `.mcp.json` in the worktree directory. Each agent's MCP server points at their personal database.

## Hybrid Search

Retrieval at session start combines two strategies:

1. **Vector similarity** — brute-force cosine similarity over serialised float32 embeddings generated by Ollama's `nomic-embed-text` model. Runs against all episodes with embeddings.
2. **BM25 keyword search** — SQLite FTS5 indexes on `facts_fts`, `episodes_fts`, and `beliefs_fts`. Uses Porter stemming.

Scores are fused: `finalScore = vectorWeight * vectorScore + keywordWeight * keywordScore` (default weights: 0.5/0.5). The top-K results are loaded, deduplicated, and ranked.

After keyword results are collected, entity mentions are extracted from the ticket description and file paths. The graph is traversed up to 2 hops from mentioned entities to find related facts.

## Beliefs with Temporal Decay

Beliefs decay over time using exponential decay with a 30-day half-life:

```
decayedConfidence = confidence * exp(-0.693/30 * daysSinceLastActivity)
```

"Last activity" is the most recent of: creation time, last confirmed time, or last accessed time. This means:

- A belief confirmed yesterday barely decays
- A belief last confirmed 30 days ago drops to ~50% of its stored confidence
- A belief last confirmed 60 days ago drops to ~25%

The minimum confidence floor is 0.1 — beliefs never fully disappear, they just become very quiet.

The `DecayBeliefs` function runs batch decay across all beliefs, updating only those whose confidence has changed by more than 0.01.

## Retrieval Strengthening

Every time an agent recalls a belief (via `TopBeliefs` for chat or via the `recall` MCP tool), the belief's access count is incremented and `last_accessed_at` is updated:

```go
_ = factStore.RecordBeliefAccess(ctx, belief.ID)
```

The access count provides a boost to the retrieval score:

```
accessBoost = 1.0 + min(accessCount * 0.05, 1.0)
decayedConfidence = confidence * decay * accessBoost
```

This caps at 2x, preventing runaway scores. Frequently recalled beliefs stay relevant longer. Beliefs that are stored but never retrieved naturally fade.

## Emotional Salience

Beliefs formed from failed sessions get higher initial confidence:

| Session Outcome | Multiplier |
|----------------|------------|
| Failure | 1.4x |
| Partial | 1.2x |
| Success | 1.0x |

A rejected PR or a failed deployment creates stronger memories because the cost of repeating the mistake is higher.

## Cross-Pollination

When agents express strong opinions in conversation (detected by signal words: "I think", "we should", "always", "prefer", etc.), the conversation engine stores the opinion as a belief with moderate confidence (0.4) in the agent's personal fact store.

If a belief becomes significant — high confidence (>= 0.6) or multiple confirmations (>= 2) — AND it mentions project-level concepts (module, architecture, pattern, dependency, etc.), it propagates to the shared project knowledge graph.

Propagation uses **confirm-or-create semantics**: FTS5 keyword matching finds similar existing beliefs. If found, the existing belief gets confirmed (bumping its confidence). If not found, a new belief is created. This prevents duplicates while reinforcing shared understanding.

## Concerns

The `ConcernTracker` (`internal/orchestrator/concerns.go`) stores things agents notice but have not verified. Concern signals are detected from phrases like:

- "worried about"
- "should check"
- "need to verify"
- "might break"
- "concerned about"

Concerns are extracted from both conversation messages and session transcripts. During idle time, `InvestigateConcerns` gives the agent a `DirectSession` to actually look into their top unresolved concern using `gh` commands. Findings are posted to `#engineering` and the concern is marked resolved.

## Seance

When a ticket is reassigned (because the previous agent failed or made partial progress), the `BuildSeanceContextFull` function assembles context from three sources:

1. **Project episodes** — previous sessions on this ticket by other agents, filtered to exclude the current agent's own work
2. **Session handoffs** — status, summary, blockers, git branch, and git state from previous sessions
3. **Cross-agent beliefs** — FTS5 keyword search across all agents' belief stores for beliefs mentioning the ticket ID

This context is injected into the implementation prompt so the new agent knows what the predecessor found, where they got stuck, and what approaches were tried.

## Handoffs

The `HandoffStore` (`internal/pipeline/handoff.go`) captures the state when a session ends:

| Field | Description |
|-------|-------------|
| `ticket` | The ticket being worked on |
| `agent` | Which agent worked on it |
| `status` | `completed`, `failed`, or `partial` |
| `summary` | Truncated transcript (500 chars max) |
| `remaining` | What is left to do |
| `git_branch` | Branch name (e.g. `feat/JAM-42`) |
| `git_state` | `clean`, `dirty`, or `conflicting` |
| `blockers` | What prevented completion |

Handoff context is injected into fix-up sessions too, so engineers addressing review feedback know what happened in the previous implementation session.

## Findings

The `PersistFindings` function scans session transcripts for discovery keywords ("discovered", "found", "unexpected", "gotcha", "warning", "blocker"). When found, it extracts the relevant sentences and posts them as a comment on the Linear ticket via the PM's `DirectSession`. This creates a permanent record of what was learned during implementation.

## Orchestrator-Driven Memory Flush

After each session, the orchestrator runs a fallback memory extraction (`FlushSessionMemory`). The PM agent receives a prompt to analyse the session transcript and extract:

- **Facts**: specific things learned about code, modules, or processes
- **Beliefs**: general opinions formed from the experience
- **Entities**: modules, files, or concepts encountered

The PM responds with JSON, which is parsed and stored in the working agent's personal knowledge graph. This is a safety net — agents should actively manage their own memory using MCP tools during the session. The orchestrator extraction catches anything they missed.

## Personality Regeneration

Every N sessions (configurable via `personality_regen_every`, default 20), `GeneratePersonalitySummary` reads the agent's top beliefs and generates a "Learned Beliefs" section categorised by strength:

- **(strong)** — confidence >= 0.8
- **(moderate)** — confidence >= 0.5
- **(weak)** — confidence < 0.5

This summary is suitable for appending to the base personality file, giving the agent earned knowledge grounded in actual experience with the codebase.

## Architecture Decisions

The Tech Lead's architectural guidance is stored as beliefs. When the Tech Lead posts a `DECISION:` statement during discussions or architecture reviews, the decision text is stored with confidence 0.7 in the Tech Lead's fact store. PM tie-breaking decisions are stored the same way. These surface in future memory retrieval when relevant topics come up.
